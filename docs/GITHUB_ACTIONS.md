# GitHub Actions Workflows

This document explains the automated workflows set up for the TinySprites project.

## Overview

The project includes three main workflows that automate building, testing, and releasing:

1. **Build & Size Check** - Runs on every push/PR
2. **Size Monitoring** - Comments on PRs with size analysis
3. **Auto-Release** - Creates releases from version tags

## Workflow Files

### 1. `.github/workflows/build.yml`

**Triggers:** Push to main/master/develop, PR to main/master

**What it does:**

- Installs dependencies with `npm ci`
- Builds the project with `npm run build`
- Calculates and displays file sizes
- Uploads build artifacts for 7 days
- Shows size warnings if files exceed limits

**Output example:**

```
📊 Build completed successfully!
==================================
Decoder: 3,300 bytes
Encoder: 7,400 bytes
Total: 10,700 bytes
==================================
✅ All checks completed!
```

### 2. `.github/workflows/size-check.yml`

**Triggers:** PR to main/master, manual dispatch

**What it does:**

- Builds and calculates sizes
- Comments on PRs with size analysis
- Updates existing comments if already present
- Shows size budget compliance (≤20KB total)

**PR Comment example:**

```markdown
## 📊 Size Analysis for PR #123

### Bundle Sizes

- **Decoder**: 3,300 bytes
- **Encoder**: 7,400 bytes
- **Total**: 10,700 bytes

### Size Impact

✅ Within size budget (≤20KB)

---

_Auto-generated by GitHub Actions_
```

### 3. `.github/workflows/release.yml`

**Triggers:** Push to main/master, version tags (v\*)

**What it does:**

- Builds the project
- Calculates file sizes
- Creates GitHub releases (only on tags)
- Uploads dist files as release assets
- Includes comprehensive release notes

**Release Notes include:**

- Bundle sizes and total
- File descriptions
- Build information (timestamp, commit, branch)
- Usage examples for ES modules and CommonJS

## Size Limits

The workflows enforce these size budgets:

- **Decoder**: ≤10KB
- **Encoder**: ≤10KB
- **Total**: ≤20KB

Warnings are shown if limits are exceeded.

## Creating Releases

### Automatic Release Process

1. **Tag your version:**

   ```bash
   git tag v1.1.0
   git push origin v1.1.0
   ```

2. **Workflow automatically:**
   - Builds the project
   - Calculates sizes
   - Creates GitHub release
   - Uploads dist files
   - Generates release notes

### Manual Release

You can also:

1. Go to GitHub → Releases → "Draft a new release"
2. Select your tag
3. The workflow will populate release notes with size info

## Local Development

### Size Checking

```bash
# Check sizes of existing build
npm run size:check

# Build and check sizes
npm run size:full

# Quick size check (builds first)
npm run size
```

### Scripts Added

- `npm run size:check` - Check existing build sizes
- `npm run size:full` - Build then check sizes
- `npm run size` - Original size command

## Workflow Dependencies

The workflows use these GitHub Actions:

- `actions/checkout@v4` - Check out code
- `actions/setup-node@v4` - Setup Node.js with caching
- `actions/upload-artifact@v4` - Upload build artifacts
- `softprops/action-gh-release@v1` - Create releases
- `actions/github-script@v7` - Comment on PRs

## Troubleshooting

### Common Issues

1. **Build fails**: Check Node.js version compatibility
2. **Size check fails**: Ensure dist/ files exist
3. **Release not created**: Verify tag format (v\*)
4. **PR comments not working**: Check workflow permissions

### Debugging

- Check Actions tab for workflow logs
- Verify file paths in dist/ directory
- Ensure proper Node.js version (18+)
- Check npm cache and dependencies

## Customization

### Modifying Size Limits

Edit the size constants in:

- `scripts/check-size.js` - Local limits
- Workflow files - CI/CD limits

### Adding New Checks

1. Add new step to workflow
2. Update size calculation if needed
3. Modify output format as desired

### Changing Triggers

Modify the `on:` section in workflow files:

```yaml
on:
  push:
    branches: [main, develop, feature/*]
  pull_request:
    branches: [main]
  workflow_dispatch: # Manual trigger
```

## Security

- Workflows run on `ubuntu-latest`
- Use `npm ci` for reproducible builds
- Artifacts are retained for 7 days only
- No sensitive data in logs or outputs
